# 图片处理策略

本项目按“是否需要用户下载”来决定图片格式。展示型系统图片优先减小体积；用户照片优先兼容手机相册、分享和下载。

## 上传架构

- 生产环境启用 COS 时，小程序直传 COS，后端只负责生成上传对象 key 和签名。
- COS 永久密钥只保存在后端，不下发到小程序。
- 小程序使用 `cos-wx-sdk-v5` 上传，SDK 每次请求签名时调用后端 `/api/uploads/cos-authorization`。
- 后端只签当前登录用户自己的上传 key，头像和评价照片分别限制在各自的 COS 前缀下。
- 本地未启用 COS 时，小程序回退到后端上传接口，方便开发环境继续使用本地文件存储。
- 上线前需要在微信小程序后台配置 COS 域名为合法 request/uploadFile 域名，并在 COS Bucket 配置允许小程序域名发起上传请求。

## 头像

- 头像属于系统展示图，不提供下载原图。
- 生产环境启用 COS 时，头像由小程序直传 COS，并通过数据万象上传时处理保存为 WebP。
- 当前头像处理规则为 `imageMogr2/auto-orient/thumbnail/512x512>/format/webp/quality/80`。
- 头像最终对象名使用 `.webp`，小程序头像展示组件需要开启 WebP 解码。
- 本地未启用 COS 时，头像上传保留原 JPEG 或 PNG 文件，作为开发 fallback。

## 普通照片

- 普通照片属于用户可能下载到本地的资产，不使用 WebP 作为唯一存储格式。
- 普通用户上传后保存压缩 JPG，用于展示、查看和普通下载。
- 相册展示图统一保存到 `uploads/session-album/display/...jpg`。
- 生产环境启用 COS 时，相册图由客户端 SDK 直传，并通过数据万象上传时处理保存为压缩 JPG。
- 当前相册展示图处理规则为 `imageMogr2/auto-orient/thumbnail/2048x2048>/format/jpg/quality/85/strip`，用于限制最长边、统一 JPG 并去除 EXIF。
- 客户端 PUT 的 `Content-Type`/`Content-Length` 是源文件事实；数据库最终保存的 MIME、字节数、宽高来自处理后 JPEG 的 HEAD → ImageInfo → HEAD，禁止把源文件字节数与处理后字节数比较。
- 上传使用同 key `Pic-Operations`：`fileid` 必须等于当前对象 key，并配合禁止覆盖头；最终确认不读取完整图片正文。
- 生产读取使用私有 Bucket 的五分钟签名 URL，缩略图、预览和下载分别签名；签名前必须完成现有隐私过滤。
- 本地未启用 COS 时，后端使用同等参数生成展示 JPG，作为开发 fallback。

## 会员照片

- 会员照片保存压缩 JPG 用于默认展示。
- 会员照片额外保存原图，用于“下载原图”权益。
- 原图权益只对上传时已保留原图的照片生效；普通用户历史上传且未保留原图的照片，后续开通会员也无法恢复原图。

## 成本与实现原则

- 头像走 WebP，因为不进入保存到相册链路，兼容风险低，能节省存储和流量。
- 照片走 JPG，因为手机相册、下载、转发和跨应用查看更稳。
- 不对同一张图片反复做下载时转码；需要长期复用的派生图应在上传时或后台任务中生成并保存。
- 当前已落地头像 WebP 通路和相册展示 JPG 通路；会员原图保留作为后续照片权益实现。
