# 车局生命周期与删除决策记录

最后更新：2026-07-03

本文是车局“取消、删除、退出车头、相册归属”的权威决策记录。后续实现、排查数据问题、设计管理后台时，以本文为准。

## 核心结论

1. 车局按微信群理解：车就是群，车头就是群主，相册照片属于这个群。
2. 车局没有“下架”状态。
3. 车局只要还存在，任何时候至少有一个车头；不能出现无车头车局。
4. 管理员可以查看车局，并可通过二次确认的“强制删除”清理测试或异常车局；管理员不能下架或恢复车局，也不能把强制删除伪装成普通用户取消。
5. 车头取消自己发起的车时，本质是解散群并删除车局；如果不满足删除条件，则不能把车变成“已下架”或“已取消”来绕过。
6. 如果已有其他玩家上车，车头不能删除车局，只能退出车头；系统把车头转给下一位已上车成员。
7. 如果车上只剩当前车头，且相册已有照片，不能删除车局；必须先处理完所有照片，避免出现无主照片。
8. 相册照片属于具体车局。删除车局时不能留下没有车局归属的照片记录或存储对象。

## 微信群类比

车局的产品心智是微信群：

- 群主退出时，如果群里还有人，群主身份转给下一个成员。
- 群里只剩群主时，取消车才等同于解散群。
- 群相册还有照片时，不能解散群；必须先清空群相册。
- 群不存在后，不能留下仍然可访问但没有所属群的照片。

因此，“下架车”“隐藏车”“无主车”“无主照片”都不是合法状态。

## 术语

| 词 | 含义 |
| --- | --- |
| 车局 | `sessions` 共享对象，等价于微信群，承载座位、报名、聊天、相册、记录等能力。 |
| 车头 | 当前 `sessions.organizer_user_id` 对应用户，等价于群主。 |
| 已上车成员 | 座位状态为 `confirmed` 或 `locked`，且 `confirmed_user_id` 不是当前车头的用户。 |
| 最后一个人 | 没有其他已上车成员可继承车头。车头自己的占位不算“其他成员”。 |
| 相册照片 | `session_album_photos` 中仍归属于该车局的照片，包括前端可见照片和后端仍保留所有权的照片记录。 |

DM、NPC 绑定用户不用于判断“是否已有玩家上车”。他们可以是车局参与角色，但不能让“删除车局”被误判为有玩家可继承。

## 用户侧规则

### 我的发车列表

车头在“我的发车”里只看到两类危险操作：

| 条件 | 按钮 | 行为 |
| --- | --- | --- |
| 没有其他已上车成员，且没有相册照片 | 取消 | 二次确认后物理删除车局和可安全清理的关联数据。 |
| 有其他已上车成员 | 退出 | 二次确认后退出车头，车局转给下一位已上车成员。 |
| 没有其他已上车成员，但有相册照片 | 不允许取消删除 | 提示先删除所有照片；不转让、不删除、不创建下架状态。 |

### 车头管理页

车头管理页必须和“我的发车”保持同一套规则：

- 可以删除时，显示“取消车/取消本车”，并说明删除会清理车局相关数据。
- 有其他已上车成员时，隐藏“取消本车”，提示“已有玩家上车，不能取消删除；请退出车头，系统会转给下一位已上车成员。”
- 没有接任者但有相册照片时，隐藏“取消本车”，提示“相册已有照片，需先删除所有照片后才能删除车局。”
- 退出车头永远不能制造无车头车局；没有接任者时退出失败，并提示当前已经是最后车头。

### 我参与的车

玩家从“我参与的车”删除入口，只删除自己的列表入口或参与关系展示，不删除共享车局，不影响其他车友。

## 管理员规则

管理员后台的车局能力分为排查查看和强制清理：

- 可以查看全量车局，用于排查数据和客服定位。
- 可以强制删除测试、异常或找不到用户的车局。
- 强制删除必须是 `system_admin` 专用入口，并要求前端二次确认。
- 强制删除会删除车局及其数据库关联数据，包括座位、报名、聊天、记录、相册记录和隐私设置。
- 第一版强制删除只保证数据库记录清理；上传对象清理由后续对象回收流程处理。
- 不能下架或恢复车局。
- 不能把强制删除接到普通用户取消入口上。

店家、剧本等资料可以有上下架和删除流程；车局不复用资料管理的上下架模型。

## 删除条件

车头发起删除车局时，后端必须在事务中重新判断，不能只相信前端显示状态。

删除允许条件：

1. 当前用户是车头。
2. 不存在其他已上车成员。
3. 不存在仍归属于该车局的相册照片。
4. 删除发生后，这个车局不再存在；只要车局存在，就必须仍有车头。

任一条件不满足时返回明确错误：

| 条件 | 建议错误码 | 文案 |
| --- | --- | --- |
| 用户不是车头 | `FORBIDDEN` | 只有车头可以执行这个操作。 |
| 有其他已上车成员 | `SESSION_HAS_ONBOARD_MEMBERS` | 已有玩家上车，不能取消删除。请退出车头，系统会转给下一位已上车成员。 |
| 有相册照片 | `SESSION_HAS_ALBUM_PHOTOS` | 相册已有照片，不能取消删除。请先删除所有照片。 |

## 相册归属规则

相册照片不能因为删除车局而变成无主数据。

第一版采用保守策略：

1. 如果车局有任何仍归属于它的相册照片，禁止删除车局。
2. 用户必须先在相册中删除照片，再回到车头管理页删除车局。
3. 删除照片不是“隐藏照片”：应删除照片标签、照片记录，并清理本地或 COS 上传对象。
4. 后端删除车局时不能单纯 `DELETE FROM session_album_photos` 来绕过检查。

后续如果要支持“一键删除车和照片”，必须先设计照片对象删除、失败重试、审计日志和回滚策略；不能在当前取消车接口里隐式吞掉照片。

## 车头转让规则

当已有其他已上车成员时，车头取消不能删除车局，只能退出车头。

转让候选人的优先来源：

1. 已确认或已锁定座位的玩家。
2. 其他被产品明确允许接任车头的绑定用户。

删除判断只看第 1 类“已上车玩家”。DM/NPC 绑定不应阻止“最后车头删除空车”，也不应让系统误以为有普通玩家可继承。

## 前端文案原则

按钮文案要直接反映实际行为：

- 会删除车局时，叫“取消”或“取消本车”。
- 会转让车头时，叫“退出”或“退出车头”。
- 不能执行时，不展示危险按钮，展示原因和下一步。
- 不使用“下架”“隐藏”“恢复”描述车局。

确认弹窗要说清楚后果：

- 删除车局：`取消后这辆车会被直接删除，座位、报名、聊天和可安全清理的数据会一起删除。`
- 退出车头：`已有玩家上车，不能取消删除。退出车头后，系统会转给下一位已上车成员。`
- 有照片拦截：`相册已有照片，不能取消删除。请先删除所有照片，避免留下无主照片。`

## 数据与实现映射

| 规则 | 后端映射 |
| --- | --- |
| 我的发车只显示当前车头创建的车 | `GET /api/users/me/sessions` |
| 其他已上车成员计数 | `confirmed/locked session_seats`，排除当前车头 |
| 用户取消车 | `PATCH /api/sessions/:id/cancel`，满足条件才物理删除 |
| 用户退出车头 | `PATCH /api/sessions/:id/organizer/leave` |
| 管理员查看车局 | `GET /api/admin/sessions` |
| 管理员强制删除车局 | `DELETE /api/admin/sessions/:id`，只允许 `system_admin`，复用车局树删除顺序 |
| 相册照片拦截 | 删除车局前检查 `session_album_photos` 是否仍有归属照片 |
| 删除相册照片 | 删除 `session_album_photo_tags`、`session_album_photos`，并清理上传对象 |

## 与旧设计的关系

本文替代以下旧方向：

- “车局个人关系下架”中的车头发起关系下架规则。
- “管理员车局管理”中的管理员取消、删除、恢复规则。
- “取消车后保留 cancelled 状态再删除”的流程。

玩家删除自己参与列表入口的能力仍保留，但它只影响个人入口，不影响共享车局。

## 回归检查建议

每次改动这块逻辑时至少检查：

1. 管理员后台没有车局下架或恢复按钮，只有明确危险文案的强制删除按钮。
2. 车头取消空车会删除车局。
3. 有其他已上车玩家时，取消被拒绝，退出车头会转让。
4. 车头自己的座位不阻止删除。
5. DM/NPC 绑定不被当成普通上车玩家。
6. 有相册照片且没有接任者时，删除被拒绝。
7. 删除车局不会留下无车局归属的相册照片。
