# 相册优先生命周期设计

日期：2026-07-03

## 结论

小程序和网页版本采用同一套“车局生命周期主行动”设计：发车前，车局仍是招募、选角、分享和车头管理工具；发车后，同一辆车自动把相册、上传、标注和车友记录提到主位置。

本设计选择 A 路线：不新增公开相册广场，不增加新的顶层相册导航，不扩大相册隐私边界。相册作为具体车局的发车后主功能存在。

## 背景

当前产品主链路已经跑通“建车 -> 分享 -> 玩家申请 -> 审核锁座”。相册页、隐私页、车友记录和网页小程序相册入口也已经存在，但入口层级仍偏向发车前：车详情第一屏主要展示基础信息、座位、分享和管理，相册只是一个按钮。

用户新的产品判断是：拼车之前，车是主要功能；发车之后，相册才是最重要的功能。设计目标是让玩家能自然回味过程，同时保持 KISS 原则。

## 设计原则

1. 同一辆车，不同阶段换主行动。
2. 不把相册做成公开内容平台。
3. 不新增复杂数据库状态；优先用已有 `session.status`、`start_at`、相册数量和记录数量派生展示阶段。
4. 小程序和网页版本规则一致，视觉可不同，但主次、文案和权限一致。
5. 成员隐私继续优先于管理员便利；管理员不能绕过同车成员可见性。

## 生命周期展示阶段

底层车局状态仍以现有状态为主：

- `draft`
- `recruiting`
- `locked`
- `cancelled`

用户主路径只理解两个阶段：

| 阶段 | 判定 | 主目标 | 主行动 |
| --- | --- | --- | --- |
| 发车前 | `start_at > now` 且未取消 | 成车和到场 | 选择角色、分享、车头管理 |
| 已发车 | `start_at <= now` 且未取消 | 相册、照片、标注和记录 | 打开相册、上传照片、写记录、回看相册 |

`cancelled` 是异常分支，不进入主路径。取消后只保留必要的车局信息和已有可见内容入口，不再引导上传。

“回味”不作为状态存在，只作为已发车后的产品语气：

- 刚发车后还没有照片：强调“相册开放”和“上传第一张”。
- 已有照片或记录：强调“回看相册”和“车友记录”。
- 内容缺失时：强调“等你补传”，不新增阶段。

## 阶段计算

推荐前端工具函数：

```text
isStarted = Date.parse(start_at) <= Date.now()
hasAlbumContent = visible_photo_count > 0 || review_count > 0

if cancelled -> cancelled
else if !isStarted -> pre_start
else -> post_start
```

`hasAlbumContent` 只控制已发车后的文案和按钮优先级，不控制阶段。

如果列表 API 暂时没有相册数量，列表只用 `start_at` 计算：

- 未发车：展示招募行动。
- 已发车但无数量：展示“打开相册”。
- 进入详情或相册后再展示准确照片数量、隐藏数量和待标注提示。

## 小程序设计

### 我的日程

车局卡片继续保留现有日程结构，但主按钮随阶段变化：

- `pre_start`：主按钮为“选择角色”或“管理”，次按钮为“分享”。
- `post_start`：主按钮为“打开相册”，次按钮为“上传照片”或“写记录”。

已发车后的按钮文案按内容轻微变化：

- 没有照片和记录：主按钮可写“上传照片”或“打开相册”。
- 已有照片或记录：主按钮可写“回看相册”。
- 用户已有记录：次按钮可写“编辑记录”。

卡片文案：

- 未发车：`招募中 · 2 个空位`
- 已发车：`已发车 · 相册开放`
- 已有内容：`已发车 · 36 张可见`
- 内容为空：`已发车 · 等你补传`

### 车详情

详情页按阶段调整第一屏：

- 未发车：保持当前招募详情，展示基础信息、座位、选择角色、分享和车头管理。
- 已发车：第一屏增加相册摘要区，主按钮为“打开相册”或“上传照片”；基础信息、座位和价格下沉为“车局信息”。

已发车详情页可以用“回看”语气，但不展示额外状态：

- 无内容：`相册已开放，上传那天的第一张照片。`
- 有内容：`一起回看这场车。`
- 有我的照片：`有你出现的照片已整理出来。`

### 车局相册

相册页继续作为具体车局页面存在，不提供相册总列表。

发车后同车成员进入相册后：

- 可上传照片。
- 可标注照片中的车友、DM/NPC、NPC 角色。
- 可进入隐私设置。
- 可按全部、我上传的、有我、待标注过滤。

未发车进入相册时提示：`相册会在发车后开放`。

非同车成员进入相册时提示：`车局相册发车后仅同车成员可查看`。

## 网页版本设计

网页小程序和小程序保持同构：

- 不把 `SessionAlbumWorkspace` 恢复为独立左侧导航。
- 从网页小程序的车详情、我的列表或已发车车局卡片进入相册。
- 未发车时操作区以选择角色、复制分享、车头管理为主。
- 已发车后操作区以车局相册、上传照片、写记录为主。
- 网页端同样使用相册开放时间和成员权限判断。

网页版本可以继续沿用当前后台工作台视觉，但文案应避免“后台相册管理”感，改成用户侧语义：`车局相册`、`上传照片`、`标注成员`、`隐私设置`。

## 数据和接口

优先复用现有接口：

- `GET /api/sessions/:id`
- `GET /api/sessions/:id/album`
- `GET /api/sessions/:id/album/people`
- `GET /api/sessions/:id/reviews`
- `GET /api/sessions/:id/review`
- `GET /api/users/me/sessions`
- `GET /api/users/me/signups`

为提升列表体验，后续可以在“我的发车”和“我参与的车”返回轻量派生字段：

- `visible_photo_count`
- `review_count`
- `has_my_review`
- `album_open`

这些字段只用于展示和按钮文案，不改变相册权限。真正进入相册时仍以相册接口权限校验为准。

## 错误和边界

- 未登录用户点击相册、上传或隐私设置时，先触发登录。
- 未发车用户点击相册时，不进入相册页，只提示发车后开放。
- 非成员访问相册返回 403，并展示同车成员说明。
- 相册数量加载失败时，不阻断车详情，只降级为“相册已开放”。
- 车局取消后不引导上传；已存在记录和照片仍按权限可看时，仅保留查看入口。

## 测试与验收

1. 未发车车局在小程序“我的”和详情页中仍以选择角色、分享、管理为主。
2. 到达 `start_at` 后，同一车局主按钮变为打开相册或上传照片。
3. 相册已有照片或记录后，列表和详情仍显示“已发车”，但按钮可变为“回看相册”。
4. 已发车但暂无照片时，展示补传引导，不显示招募主行动。
5. 网页小程序车详情遵循同样阶段规则。
6. 未发车访问相册只提示，不进入相册。
7. 非同车成员不能查看相册内容。
8. 管理员不能绕过成员隐私查看不可见照片。
9. 现有相册过滤、上传、标注和隐私设置不被破坏。

## 非目标

- 不做公开相册广场。
- 不做跨车局相册瀑布流。
- 不新增复杂相册审核流。
- 不改变照片隐私模型。
- 不承诺平台长期永久保存所有原图。

## 自检

- 本设计没有把相册变成公开内容平台。
- 本设计没有新增硬数据库状态，也没有新增第三个展示状态，避免状态机膨胀。
- 用户主路径只有发车前和已发车两个阶段，理解成本最低。
- 小程序和网页版本有同构规则。
- 权限和隐私规则延续现有相册整改方向。
