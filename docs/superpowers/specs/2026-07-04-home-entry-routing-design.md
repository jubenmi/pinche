# 首页车局分流设计

日期：2026-07-04

## 结论

小程序首页从品牌入口页改为“按用户成熟度分流”的车局入口，但界面只保留两类：

1. 首车入口：未登录用户和已登录但没有任何车局的用户共用。
2. 车局日历：已登录且已有车局的用户看到，顶部提供“发车”按钮。

本设计不新增公开车局广场，不改变发车流程本身，不改变分享落地路径。首页继续保留维护态和微信分享能力。

## 背景

当前首页是品牌图加“创建 / 我的”两个入口，真正的回访价值在“我的拼车日程”里。用户提出希望登录后首页直接展示车局，并把创建入口放到上方。讨论后确定冷启动需要按认证和车局数据判断，但未登录用户和登录无车用户可以共用同一个首车入口：

- 未登录用户不应看到空日历，也不需要单独一套登录页。
- 已登录但没用过的用户同样进入首车入口。
- 已有车局的用户应直达日历和待处理事项。

这个方向把首页从“入口选择”改为“下一步工作台”。

## 目标

1. 降低首页冷启动：未登录和没车用户共用首车入口。
2. 提高回访效率：已有车局的用户打开首页即可看到发起、参与、待审核和已发车车局。
3. 保留创建效率：日历顶部始终有明确“发车”按钮。
4. 复用现有流程：点击发车进入当前 `/pages/session/create`，后续仍是选店、选本、选角色、开本设置和分享。
5. 保留维护态和分享能力：后端维护时首页仍展示维护页；微信分享仍落到 `/pages/index/index`。

## 非目标

- 不做公开车局列表或发现页。
- 不自动创建草稿车局。
- 不改变发车流程页面顺序。
- 不改变报名、审核、相册、车友记录和隐私规则。
- 不新增后端接口，除非实现时发现现有接口性能无法接受。

## 用户分流

首页加载后按以下顺序判断：

```text
if backend maintenance:
  show maintenance state
else if no valid user or no token:
  show first-session entry state
else:
  load my sessions and my signups
  if combined calendar count == 0:
    show first-session entry state
  else:
    show calendar state
```

“有车局”的判定使用“我的发车”和“我参与的车”合并结果：

- `GET /api/users/me/sessions?limit=50`
- `GET /api/users/me/signups`

只要任一接口返回可展示条目，就进入日历状态。被个人删除、隐藏或无权限的车局不应计入，因为这些接口本身已经过滤用户当前可见关系。

## 状态设计

### 首车入口

目标：把未登录用户直接视作“尚未确认是否有车”的首车用户，避免出现单独登录页或空日历，同时保留登录后改判为日历的能力。

界面：

- 标题：`发起第一辆车`。
- 说明文案：`先选店家和剧本，几步就能生成分享卡片。`
- 主按钮：`开始发车`。
- 辅助文案：`之后你的发车、报名和相册都会自动汇总到首页。`

行为：

- 未登录用户点击 `开始发车` 时先调用现有 `ensureLoggedIn()`。
- 登录成功后必须重新加载“我的发车”和“我参与的车”，再决定下一步。
- 如果登录后发现用户已有车局，进入车局日历，不自动进入发车流程。
- 如果登录后仍没有任何车局，调用 `clearCreateFlow()`，再 `navigateTo("/pages/session/create")`。
- 已登录且确认没有车局的用户点击 `开始发车` 时，直接清空创建流程并进入 `/pages/session/create`。
- 登录取消或失败时留在首车入口，展示轻量错误提示。
- 如果加载我的车局失败，不把用户误判为没车；展示加载失败文案并提供重试。

### 车局日历状态

目标：把已有车局用户的首页变成回访工作台。

界面：

- 标题使用 `我的车局`，让首页语义从个人资料页转为车局工作台。
- 副文案：`X 场车局 · 最近更新刚刚`。
- 顶部主按钮：`发车`。
- 管理员入口和退出入口弱化，不能与 `发车` 抢主行动。
- 继续保留现有筛选：全部、发起、参与、待处理。
- 继续保留按日期聚合、归位、日期选择、下拉刷新和加载更多。

行为：

- 点击 `发车` 时调用 `clearCreateFlow()`，进入 `/pages/session/create`。
- 点击车局卡片复用现有逻辑：
  - 我发起的车进入车头管理。
  - 已发车且可看的参与车进入相册。
  - 可写记录的车进入写记录。
  - 其他车进入详情。
- 删除、退出车头、隐藏参与关系继续遵守现有规则。

## 信息架构

首页成为主入口：

```text
/pages/index/index
  maintenance
  first-session entry
  calendar
    create button -> /pages/session/create
    session card -> manage/detail/album/review
```

“我的”页本次继续保留为同一套车局日历入口，并和首页共享车局日历组件。后续是否弱化或移除独立“我的”入口，另开设计处理。

实现时优先抽出 `SessionCalendar` 组件，避免把 `mine/index.vue` 的日历逻辑复制到首页。首页负责分流和状态壳，组件负责日历展示和卡片行为。

## 数据和错误处理

### 登录判定

必须同时满足：

- `getCurrentUser().user` 存在。
- `getToken()` 存在。

如果有用户缓存但没有 token，应清理认证并回到首车入口。

### 加载中

登录用户进入首页后，以及未登录用户登录成功后，在判定“没车”或“有车”之前需要展示加载态：

- 文案：`正在整理你的车局...`
- 加载态不能闪成首车入口。

### 接口失败

如果我的发车或我参与接口失败：

- 401：清理登录，回到首车入口，并提示 `登录已过期，请重新登录。`
- 维护或网络失败：展示错误状态和重试按钮。
- 其他单接口失败：不误判为没车。可显示 `车局加载失败，请稍后重试。`

只有两个接口都成功，且合并后数量为 0，才允许已登录用户进入首车入口或继续进入发车流程。

### 分享落地

首页分享仍打开 `/pages/index/index`。

分享进来的用户看到的状态取决于当前账号：

- 未登录：首车入口。
- 登录但没车：首车入口。
- 登录且有车：自己的车局日历。

本次不为首页分享新增公共介绍页。如果后续需要面向完全陌生用户做增长页，应另开设计。

## 实现边界

建议实现拆分：

1. 抽日历展示和行为为组件，输入 `sessions`、`signups`、加载状态和操作回调。
2. 首页新增分流状态机，复用现有维护检测、分享菜单和认证工具。
3. “我的”页改为复用同一日历组件，减少重复逻辑。
4. 静态检查扩展首页断言：首车入口、日历、发车按钮、维护态、登录后重判和分享回调都存在。

不建议直接把 `mine/index.vue` 整段复制进首页。复制会让后续相册、记录、删除和筛选逻辑出现两份实现。

## 测试与验收

1. 后端维护时，首页仍优先展示维护态，不发起我的车局请求。
2. 无 token 用户进入首页，看到首车入口，不展示空日历。
3. 未登录用户点击 `开始发车` 后先登录，登录成功后重新加载我的车局。
4. 登录后如果发现已有发起车局或参与车局，进入日历，不自动进入发车流程。
5. 两个我的车局接口都成功且均为空时，展示首车入口。
6. 已登录无车用户点击 `开始发车` 会清空创建流程并进入 `/pages/session/create`。
7. 任一我的车局接口失败时，不继续进入发车流程，也不展示空日历误导用户。
8. 有发起车局或参与车局时，首页展示日历。
9. 日历顶部有 `发车` 主按钮，点击后清空创建流程并进入当前发车流程。
10. 车局卡片点击、删除、退出车头、隐藏参与关系保持现有行为。
11. 首页仍注册 `onShareAppMessage`、`onShareTimeline`，并继续使用 `/pages/index/index` 分享路径。
12. “我的”页和首页的日历行为一致，不出现两套不同的卡片逻辑。

## 自检

- 本设计用同一个首车入口承接未登录和登录无车用户，不再让没车用户面对空日历。
- 本设计没有新增公开发现页，符合当前产品仍以个人车局和分享链路为核心的方向。
- 本设计没有改变创建、报名、相册和隐私模型。
- 首页维护态和分享落点被保留。
- 实现建议强调组件复用，避免首页和“我的”产生两份日历逻辑。
