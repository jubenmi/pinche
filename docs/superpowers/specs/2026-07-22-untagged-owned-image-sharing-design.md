# 本人未标注图片便捷分享设计

日期：2026-07-22

## 结论

本人上传、尚未标注的图片可以在本人确认一次公开风险后进入公开相册分享和单图分享，但系统不把它们写成“其他”标签。

未标注仍是独立的数据状态：在成员相册中只有上传者可见，不会因为允许分享而自动展示给同车成员。系统为图片单独保存“上传者已允许未标注公开分享”的授权状态；该状态只影响上传者本人发起的公开分享，不改变标签、成员可见性或其他用户的分享权限。

## 背景

当前公开分享要求媒体至少存在一个角色或场景标签。即使图片是分享者本人上传，只要尚未标注，也不能进入整册分享或单图分享。用户必须逐张进入标注流程并选择“其他”，才能使用普通风景、道具或无需人物标注的图片，操作成本与获得的分类价值不匹配。

现有“其他”并不只是展示文案。它是一个明确标签，表示“风景/主线外照片”，并会使符合条件的媒体进入同车成员的公共可见范围。直接把所有未标注图片写成“其他”会产生两类问题：

- 把“尚未处理”误记为“用户明确确认不需要人物标注”。
- 把原本仅上传者可见的图片自动扩大到同车成员可见。

因此，便捷分享必须与“其他”标签、成员相册可见性分离。

## 目标

1. 本人上传的未标注图片无需逐张选择“其他”即可用于本人发起的分享。
2. 首次公开前只进行一次批量确认，不要求逐张标注或逐张确认。
3. 未标注图片在成员相册中继续只有上传者可见。
4. “其他”继续表示用户明确选择的场景类标签，不被系统自动写入。
5. 上传者和被标注者的既有隐私设置继续作为公开分享的硬门禁。
6. 整册分享和单图分享使用同一套资格判断，避免两个入口产生不同隐私结果。

## 非目标

- 不自动识别人脸、角色或照片内容。
- 不允许分享者带出其他用户上传的未标注图片。
- 不把未标注视频纳入本次放宽范围。
- 不改变明确标注为“其他”、NPC、角色或席位的现有语义。
- 不改变未标注筛选、批量标注、下载或内容审核流程。
- 不让未标注图片自动成为微信分享卡片封面。

## 方案比较

### 方案一：自动写入“其他”标签

上传后或分享前直接创建 `other:session` 标签。实现表面上最简单，但会污染真实标签数据，并把图片自动扩大到同车成员可见。该方案不采用。

### 方案二：分享时临时把未标注视为“其他”

不写标签，仅在当前分享请求中放宽本人上传图片。它不会改变成员可见性，但如果没有独立授权状态，用户每次重新进入页面都需要再次确认；如果完全省略确认，又会让包含其他玩家但忘记标注的图片直接公开。该方案不单独采用。

### 方案三：独立保存未标注公开授权

未标注状态保持不变，首次分享时对当前候选图片进行一次批量确认，并保存专用于公开分享的上传者授权。后续分享可直接复用；成员可见性和标签数据不受影响。该方案在方便、隐私和数据语义之间最平衡，作为最终方案。

## 数据语义

### 标签状态

- `tags.length === 0` 继续表示“未标注”。
- `other:session` 只在用户主动选择“其他”并保存时存在。
- 客户端不得把已授权分享的未标注图片显示成“其他”；相册卡片仍显示“待标注”。

### 未标注公开授权

图片媒体增加独立的上传者授权状态，概念字段为 `allow_untagged_public_share`：

- 新上传图片默认值为 `false`。
- 只有图片上传者可以把它设为 `true`。
- 授权只对“该上传者以本人身份发起公开分享”生效。
- 授权不改变普通成员相册的可见性。
- 授权不赋予其他成员分享该图片的权限。
- 图片发生任何标签保存操作时清除该授权；如果最终仍是未标注，下一次公开前需要重新确认。
- 删除、审核拒绝或媒体失效后，授权随媒体一起失效。

使用独立授权而不是自动标签，可以明确区分“照片分类”和“上传者同意公开”两个事实。

## 分享资格

### 本人未标注图片

一张未标注图片只有同时满足以下条件，才能进入当前用户发起的公开候选：

1. 媒体类型为图片，不是视频。
2. `uploader_user_id` 等于分享者用户 ID。
3. 图片状态为 `active`。
4. 内容审核状态为已批准或兼容的历史已批准状态。
5. `allow_untagged_public_share = true`。
6. 上传者的 `allow_uploaded_visible = true`。
7. 图片属于当前车局。

未标注图片不依赖分享者席位标签，因为它没有任何标签；它只能依赖“本人上传 + 本人公开授权”建立分享关系。

### 已标注媒体

只要媒体存在标签，就完全沿用当前 D48/D50 规则：

- 分享者上传或标注了分享者席位，才满足关联条件。
- 上传者关闭公开时排除。
- 任一绑定真实用户的标签关闭被标注可见时排除。
- 旧数据中的席位用户关系无法验证时关闭式排除。
- 视频仍要求处理状态为 `ready`。

未标注公开授权在媒体存在标签时不参与资格判断。

### 动态复核

分享快照继续只保存媒体 ID。每次读取公开列表、图片文件或相关公开能力时重新检查媒体状态、审核状态、上传者隐私和当前标签：

- 图片后来被标注：立即切换到已标注媒体规则。
- 标签保存会清除未标注公开授权，因此再次清空标签后不会自动恢复公开。
- 上传者关闭 `allow_uploaded_visible`：已有分享中的图片立即不可见。
- 图片被删除、拒绝或失效：已有分享中的图片立即不可见。

## 用户交互

### 整册分享

当系统发现当前用户存在“本人上传、已审核、未标注且尚未授权公开”的候选图片时，分享入口显示一次批量确认：

> 本次分享将加入你上传的 N 张未标注照片。请确认其中没有不适合公开的其他玩家。未标注照片不会因此向同车成员自动展示。

操作：

- `取消`：不授权这些图片；当前已标注且合规的媒体仍可按现有规则分享。
- `确认并分享`：服务端只为本次确认范围内、仍满足资格的本人图片保存授权，然后重新生成公开分享快照。

确认按当前候选集合批量完成。以后新增的未标注图片没有授权，首次纳入分享时再次出现一次批量确认；已确认图片不重复要求逐张操作。

### 单图分享

用户在全屏预览中分享本人未标注图片时：

- 尚未授权：分享图标进入“待确认”状态，点击后显示同一隐私确认；确认成功后准备单图分享。
- 已授权：直接沿用现有单图分享准备和原生微信分享按钮。
- 非本人上传的未标注图片：保持不可分享。

确认失败、图片状态变化或服务端复核不通过时，保持关闭式失败，不使用其他图片或整册分享冒充当前图片。

### 相册展示

- 成员相册中的文案继续显示“待标注”，不显示“其他”。
- 未标注图片继续只有上传者可见。
- 批量标注和单张标注入口保持可用；标注是可选的精细分类和人物隐私工具，不再是本人图片分享的强制前置步骤。
- 可在分享入口附近使用轻量说明“未标注的本人图片确认后也可分享”，不增加逐张操作入口。

## 微信分享封面

未标注图片即使已经获得公开授权，也不得自动成为微信卡片封面。封面继续按现有安全优先级选择：

1. 分享者本人上传、明确标注本人席位且不含其他真实玩家的图片。
2. 分享者本人上传、只含明确安全场景标签的图片，例如 NPC、场景或“其他”。
3. 没有安全图片时使用现有品牌兜底封面。

这样可以让未标注图片出现在用户主动打开的公开相册内，但不在外部聊天列表或朋友圈入口中自动放大曝光。

## 服务端边界

公开资格判断必须由服务端统一实现，客户端只负责展示状态和发起确认：

- 扩展公开媒体资格函数，使空标签图片进入独立的“本人上传 + 已授权”分支。
- 明确限定该分支只接受图片。
- 整册 token 和带 `focusMediaId` 的单图 token 共用该资格函数。
- 批量授权请求中的媒体 ID 必须由服务端重新校验车局、上传者、媒体类型、审核状态和空标签状态；忽略或拒绝越界 ID，不能信任客户端计数。
- 授权与分享快照创建放在同一事务边界内，避免确认后媒体状态变化导致错误入选。
- 公开列表和文件读取继续动态复核，不只依赖签发 token 时的结果。
- 封面选择器继续要求非空安全标签，不读取未标注公开授权。

## 错误与竞态

- 确认前图片被删除、拒绝、标注或转为非 active：该图片不授权、不进入快照，其余合规图片继续处理。
- 确认范围全部失效且没有其他公开候选：返回 `ALBUM_PUBLIC_SHARE_EMPTY`。
- 非上传者提交授权：返回权限错误，不改变任何媒体。
- 客户端重复确认：操作幂等，不重复创建标签或额外记录。
- 标签保存与公开授权并发：标签事务清除授权；最终存在标签时只走已标注规则，最终为空标签且授权已被清除时不公开。
- 单图目标在准备期间失效：返回 `ALBUM_PUBLIC_SHARE_MEDIA_UNAVAILABLE`，不得回退其他媒体。
- 未标注授权失败：保留当前未标注和成员私有状态，可重试，不自动写入“其他”。

## 测试与验收

### 服务端单元测试

1. 本人上传、未标注、图片、已授权、审核通过且上传隐私开启时可公开。
2. 同一图片未授权时不可公开。
3. 他人上传的未标注图片即使出现异常授权值也不可由当前分享者公开。
4. 未标注视频即使授权值异常为真也不可公开。
5. 上传者关闭 `allow_uploaded_visible` 时不可公开，包括上传者本人发起分享。
6. 未批准、已删除或非 active 图片不可公开。
7. 图片存在标签后忽略未标注授权并执行完整标签隐私规则。
8. 保存标签会清除未标注公开授权。
9. 整册和单图分享对同一媒体给出一致资格结果。
10. 未标注已授权图片不会进入公开封面候选。

### API 与事务测试

1. 上传者可批量授权当前车局中本人上传的未标注图片。
2. 请求中混入他人图片、其他车局图片、视频、已标注或未批准图片时不会越权授权。
3. 重复授权保持幂等。
4. 授权和快照创建之间发生状态变化时，最终快照只包含事务复核通过的媒体。
5. 旧分享在隐私关闭、标签变化、删除或审核状态变化后动态失效。
6. 单图目标不合规时保持关闭式失败。

### 小程序测试

1. 普通相册仍把已授权的未标注图片显示为“待标注”，且其他成员不可见。
2. 首次整册分享包含 N 张候选时只出现一次批量确认，不要求逐张标注。
3. 取消确认后不改变标签或成员可见性。
4. 确认后可分享整册，后续同一批图片不重复逐张确认。
5. 新增未标注图片后，仅新增候选需要下一次批量确认。
6. 单图分享未授权本人图片时先确认，授权后可使用原生微信分享。
7. 非本人未标注图片保持不可分享。
8. 微信分享封面不会选用未标注图片；没有安全封面时使用品牌兜底图。

## 发布与兼容

- 数据库字段默认关闭，因此旧图片和旧客户端不会突然扩大公开范围。
- 旧客户端不发送授权请求时继续保持当前行为。
- 服务端先上线字段、授权校验和动态复核，再上线小程序确认交互。
- 回滚小程序后，已保存授权不会改变成员相册可见性；服务端如需完全回退放宽逻辑，可停止读取授权字段，标签数据不受影响。

## 决策摘要

“未标注”与“其他”保持两个不同概念。方便性通过一次批量公开授权解决，隐私通过本人上传限制、上传者隐私门禁、动态复核和安全封面限制保证。标注仍用于人物关系、成员可见性和精细隐私，但不再是本人图片公开分享的强制前置步骤。
