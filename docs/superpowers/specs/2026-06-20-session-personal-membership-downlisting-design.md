# 车局个人关系下架设计

> 状态：部分过期。车头发起的车局不再使用“个人关系下架/隐藏”；最新规则见 `docs/session-lifecycle-decisions.md`。玩家从“我参与”删除个人入口的规则仍可参考本文。

日期：2026-06-20

## 结论

车局按“车友群”理解：`sessions` 是所有车友共用的共享对象，车上的玩家看到同一份车局信息、记录墙和车内内容。用户“删除自己发车或者参与的记录”不是删除车局，也不是取消车，而是把自己和这台车的个人关系下架。

下架只影响当前用户在“我的”里的入口。车局本身继续存在，其他车友不受影响。用户之后可以通过其他车友分享的车局链接重新进入这台车；只有这台车的参与者才能进入和继续使用车友群能力。

## 背景

当前系统已经有：

- `sessions`：车局共享记录，由车头创建。
- `signups`：用户和车局/座位的参与关系。
- `GET /api/users/me/sessions`：返回我发起的车。
- `GET /api/users/me/signups`：返回我参与的车。
- `cancelSession`：取消整台车，会影响座位、报名和车内通知。

用户要的是“像微信群一样”的个人退群/隐藏入口，不是删除群，也不是群主解散群。因此不能复用取消车，也不能物理删除 `sessions` 或 `signups`。

## 目标

1. 车头可以从“我发起”下架自己发起的车局。
2. 玩家可以从“我参与”下架自己参与的车局关系。
3. 下架后，“我的”默认不再显示该入口。
4. 下架不删除共享车局，不影响其他车友，不释放座位，不取消报名，不清理评价资格。
5. 下架后，用户仍可通过别人分享的车局链接重新进入；进入后的权限仍按“是否是这台车的参与者/车头”判断。
6. 未下架的车局关系可以继续维护这台车。

## 非目标

- 不做物理删除。
- 不做车局取消、解散或清空聊天。
- 不改变其他车友的列表和权限。
- 不做公开搜索恢复入口。第一版通过分享链接重新进入。
- 不做管理员审计后台。

## 核心规则

### 共享车局

`sessions` 表示共享车局。对车局内容的修改对车上所有有权限的车友可见。车头下架个人入口不等于删除或取消车局。

### 个人关系

用户在“我的”中看到两类个人关系：

- 发起关系：用户是 `sessions.organizer_user_id`。
- 参与关系：用户在 `signups` 中有该车局记录。

下架只改变这类个人关系是否出现在“我的”列表。

### 重新进入

用户通过分享链接进入车局详情时，系统仍加载同一台 `session`。如果用户是车头或参与者，可以继续看到车友群相关能力。这个动作可以让用户重新获得“我的”入口，等价于重新链接这个群。

## 数据设计

新增一个 migration，例如 `apps/api/migrations/0012_session_membership_downlisting.sql`。

### sessions

新增：

```sql
organizer_hidden_at DATETIME NULL
```

含义：车头把这台车从“我发起”下架的时间。车局仍然存在。车头通过分享链接重新进入后可清空该字段。

### signups

新增：

```sql
user_hidden_at DATETIME NULL
```

含义：参与者把这台车从“我参与”下架的时间。报名关系仍然存在。参与者通过分享链接重新进入后可清空该字段。

## API 设计

### `PATCH /api/sessions/:id/hide`

登录后，只有车头可以调用。设置 `sessions.organizer_hidden_at = CURRENT_TIMESTAMP`。

返回下架后的车局摘要。

### `PATCH /api/sessions/:id/relink`

登录后，如果用户是车头，清空 `sessions.organizer_hidden_at`。如果用户是参与者，清空自己该车局下 `signups.user_hidden_at`。如果二者都是，则都清空。

用于分享链接进入详情页时重新链接。没有车局关系的用户不能借此加入群。

### `PATCH /api/signups/:id/hide`

登录后，只有该报名用户可以调用。设置 `signups.user_hidden_at = CURRENT_TIMESTAMP`。

返回下架后的报名摘要。

### 列表过滤

`GET /api/users/me/sessions` 默认增加：

```sql
session.organizer_hidden_at IS NULL
```

`GET /api/users/me/signups` 默认增加：

```sql
signup.user_hidden_at IS NULL
```

第一版不增加“显示已下架”参数。

## 小程序设计

### “我的”

“我发起”和“我参与”列表项增加“删除”按钮。点击后弹出确认：

- 我发起：文案说明“只会从你的列表下架，不会取消车，也不会影响其他车友。”
- 我参与：文案说明“只会从你的列表下架，不会影响这台车或其他车友。”

确认后调用对应 hide 接口并刷新列表。

### 车详情

用户从分享链接进入车详情时，详情页加载当前用户身份。如果用户已经是车头或参与者，调用 relink 接口恢复个人入口。未登录用户不自动恢复。

恢复失败不阻断查看公开详情，但需要车友权限的能力仍按原权限规则限制。

### 车头管理

下架后的车头不能再从“我的”进入管理页。通过分享链接重新进入并 relink 后，入口恢复，可以继续维护。

## 错误处理

- 未登录下架：401。
- 不是车头下架发起关系：403。
- 不是报名本人下架参与关系：403。
- 车局不存在：404。
- relink 时用户不是车头也不是参与者：403。

## 测试

增加静态回归检查脚本或扩展现有检查，覆盖：

- migration 新增 `organizer_hidden_at` 和 `user_hidden_at`。
- service 中存在发起关系下架、参与关系下架和重新链接函数。
- `listMySessions` 过滤 `organizer_hidden_at IS NULL`。
- `listMySignups` 过滤 `user_hidden_at IS NULL`。
- server 暴露 hide/relink 路由。
- “我的”页有删除确认和下架接口调用。
- 详情页有 relink 逻辑。

## 自检

- 本设计没有物理删除共享数据。
- 本设计没有把个人下架和取消车混用。
- 下架只影响当前用户的“我的”入口。
- 共享车局仍可被其他车友看到和维护。
- 分享链接重新进入只恢复已有关系，不创建新成员关系。
