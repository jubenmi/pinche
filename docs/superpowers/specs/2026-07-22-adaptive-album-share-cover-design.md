# 相册自适应分享封面设计

状态：用户已确认设计方向，等待书面规格复核

日期：2026-07-22

视觉方向：电影叙事型

## 1. 背景

当前相册公开分享封面将 1～9 张候选照片裁成等大的正方形格子，并让输出画布跟随网格自然宽高。六张照片时，三列两行得到约 3:2 的封面；微信好友/群聊分享卡片按 5:4 展示，因此会从右侧裁掉第三列的一部分。朋友圈要求 1:1，但小程序当前仍与好友分享复用同一张封面，无法同时保证两个场景。

现有票根兜底图为 278×78，也不匹配 5:4 或 1:1。

## 2. 目标

1. 好友/群聊封面固定为 5:4，任何照片数量都不得越出画布。
2. 朋友圈封面固定为 1:1，与好友封面独立排版。
3. 相册可以有 1 张、10 张或 100 张照片；封面从隐私安全候选中自动精选 1～9 张，不把总量直接映射为封面数量。
4. 视觉质量优先。弱图、近似重复图和严重不适配布局的图片可以不进入封面。
5. 延续用户确认的电影叙事方向：一个明确主角、若干叙事细节、暖色电影调性、克制间隔和短文案。
6. 动态中文必须准确、可读、可预测，不能交给生成模型书写。
7. 保留现有相册快照、隐私、审核、撤销和短期访问能力。

## 3. 非目标

- 不提供用户手工拖拽排版或逐张指定封面。
- 不在运行时调用生成式图像模型。
- 不把超过 9 张照片塞入一张分享封面。
- 不改变公开相册正文允许展示的媒体数量。
- 不在封面中展示真实姓名、联系方式、精确开场时间、上传者或人物标签。

## 4. 已确认的视觉系统

### 4.1 视觉层级

- 最强照片承担主视觉。
- 其余照片只补充人物、剧本、场景和互动细节。
- 间隔使用暖象牙色，不使用纯白硬边。
- 1～6 张时，文案放在主视觉底部的深色渐变安全区。
- 7～9 张时，小图密度较高，改用紧凑的电影字幕带，避免文字随机压住人物。
- 不添加复杂装饰、贴纸堆叠、手写体或多套字体。

### 4.2 确定性文案

封面只使用以下文案结构：

```text
本场掉落
这一晚，我是「叶辰」
《琼崖Ⅱ海角》 · 游玩相册
```

- `叶辰` 来自安全的分享角色名。
- 没有角色名时，主文案改为 `这一晚，故事没有散场`。
- 剧本名缺失时使用 `剧本待定`。
- 文字由服务端使用固定字体和 SVG 图层确定性渲染；图片生成或图片分析不得改写文案。
- 主文案允许最多两行。渲染器先测量文字宽度，再在批准字号范围内缩放；仍无法容纳时才对角色名或剧本名做单行省略。
- 好友封面主文案最小字号为画布宽度的 4.0%，副文案最小字号为画布宽度的 2.0%。朋友圈按相同视觉比例计算。

## 5. 场景输出规格

| 场景 | 画布 | 比例 | 用途 |
|---|---:|---:|---|
| `friend` | 1000×800 | 5:4 | 好友和群聊 `onShareAppMessage` |
| `timeline` | 1000×1000 | 1:1 | 朋友圈 `onShareTimeline` |

两个场景复用同一批精选照片和同一个主视觉，但分别进行布局、裁切和文字定位。禁止用裁切 5:4 成品的方式产生 1:1 成品。

## 7. 自动选图

### 7.1 隐私与资格门禁

继续使用现有公开相册规则：

1. 媒体必须属于当前分享快照。
2. 审核状态必须允许公开展示。
3. 媒体必须仍然可用且未删除。
4. 分享席位、角色关联和用户隐私设置必须仍然有效。
5. 只选择静态图片；视频仍可出现在公开相册正文，但不从视频源临时抽帧作为本期封面。

门禁失败的媒体不能进入视觉评分。

### 7.2 快照候选

`cover_media_ids` 继续保持最多 9 个 ID，不扩大 token 或数据库字段语义。快照阶段使用现有安全优先级，并增加以下确定性排序：

1. 与分享角色关联的照片优先于安全场景图。
2. 像素面积更大的照片优先。
3. 横竖方向与尚未填充的布局槽位更匹配者优先。
4. 在相同场景与角色关联下，拉开拍摄/创建顺序，避免连续近似画面占满候选。
5. 最后用创建时间和媒体 ID 保证稳定排序。

### 7.3 渲染阶段质量过滤

封面路由读取快照候选后，对实际图片缓冲执行轻量分析：

- 清晰度：使用标准化锐度分数。
- 曝光：将平均亮度处于 0.35～0.75 的图片视为理想；低于 0.08 或高于 0.92 施加最大曝光惩罚。
- 分辨率：长边达到 1600 像素时获得满分；更小图片按比例降分，但只要可安全渲染就不单独淘汰。
- 关联度：分享角色图高于安全场景图。
- 重复度：生成 64 位感知哈希；汉明距离不大于 6 的图片视为近似重复，只保留综合分更高的一张。

综合质量分归一化到 0～1：

```text
0.40 × 清晰度
+ 0.20 × 曝光
+ 0.20 × 分辨率
+ 0.20 × 角色/场景关联度
```

先去重，再保留分数不低于最高分 65% 的候选；至少保留一张。如果只有一张通过门禁的照片，无论相对质量如何都使用单图模板。

因此，大相册不一定生成 9 张封面：六张强图可能生成六图模板，只有三张足够好且不重复时则生成三图模板。

### 7.4 槽位分配

主视觉由综合分最高的照片承担。其余照片按以下裁切损失分配到槽位：

```text
cropLoss = abs(log(imageAspectRatio / slotAspectRatio))
```

布局分配器寻找总裁切损失最低的稳定排列；主视觉槽位的裁切损失权重为普通槽位的两倍。相同分数时以候选稳定顺序破同分。

单个槽位仍使用 `cover` 裁切，但焦点采用注意力优先定位，避免固定从中心切掉主要人物。系统保证每个槽位完整出现，但不承诺任意原始长宽比都能无裁切展示全部画面。

## 8. 服务端结构

不继续把选图、布局和 Sharp 合成逻辑堆在 `server.js` 中。新增小而独立的模块：

```text
apps/api/src/modules/album-share-cover/
  layouts.js      # friend/timeline 的 1～9 张归一化槽位
  selection.js    # 质量、去重、主视觉和槽位分配纯逻辑
  copy.js         # 安全文案与文字测量规则
  renderer.js     # 图片解码、裁切、合成和 JPEG 输出
```

模块边界：

- `layouts` 不读取文件或数据库。
- `selection` 接收候选元数据和分析结果，不接触 HTTP。
- `copy` 只接收已清理的剧本名和角色名。
- `renderer` 接收已授权的图片缓冲与最终渲染计划，不参与隐私判断。
- 路由继续负责 token、撤销状态、响应头和错误映射。

最终输出为 JPEG，质量 84，自动旋转并移除 EXIF。

## 9. API 与小程序数据流

### 9.1 封面接口

现有公开封面路径增加白名单参数：

```text
GET /api/session-album/public-shares/:shareId/cover?token=...&variant=friend
GET /api/session-album/public-shares/:shareId/cover?token=...&variant=timeline
```

- 缺少 `variant` 时默认 `friend`，兼容现有链接。
- 除 `friend` 和 `timeline` 外的值返回 400。
- token 继续绑定 share ID 和候选媒体摘要。
- variant 不改变访问权限，只决定经过验证后的输出画布。

分享 token 响应增加：

```json
{
  "cover_url": "...&variant=friend",
  "timeline_cover_url": "...&variant=timeline"
}
```

### 9.2 小程序

- `onShareAppMessage` 只使用好友封面 URL。
- `onShareTimeline` 只使用朋友圈封面 URL。
- 两个 URL 分别预取，不再通过单个 `albumShareImage()` 混用。
- 单媒体分享继续使用所选媒体的安全卡图，不进入本设计的多图自适应布局。
- 好友封面准备成功即可开放好友分享；朋友圈菜单只有在朋友圈封面准备成功后开放。

## 10. 缓存与稳定性

渲染结果使用服务端有界 LRU 缓存，键为：

```text
shareId + coverMediaIdsDigest + variant + layoutVersion
```

- `layoutVersion` 在布局、选图阈值、字体或文案结构改变时递增。
- 每次请求必须先完成 token、快照、席位、隐私和撤销校验，再读取缓存；撤销分享不能被缓存绕过。
- 对相同快照和版本，选图、排序、裁切与文案必须确定性一致。
- 外部响应继续使用私有缓存策略，不能让公共 CDN 绕过撤销能力。

## 11. 错误处理与兜底

1. 单张候选解码失败时跳过该候选，继续用剩余图片布局。
2. 所有候选都不可用时，封面接口返回明确的不可生成错误，不返回当前页面截图。
3. 小程序分别提供：
   - 1000×800 的好友品牌兜底图。
   - 1000×1000 的朋友圈品牌兜底图。
4. 兜底图沿用暖象牙、墨绿和电影字幕语言，不包含照片、隐私字段或联系方式。
5. 当前 278×78 的票根图不再承担相册分享封面兜底。

## 12. 验收与测试

### 12.1 纯逻辑测试

- friend 和 timeline 的 1～9 张模板均返回正确槽位数量。
- 所有槽位完全位于 0～1 归一化画布内且互不越界。
- 主视觉槽位对每个数量都唯一。
- 近似重复图片只保留质量更高者。
- 全部低质量时仍稳定保留最佳一张。
- 同一输入始终得到相同数量、主视觉和槽位顺序。
- 横图、竖图、方图混合时，槽位分配选择更低裁切损失。

### 12.2 渲染测试

- 18 个最终组合（2 个场景 × 1～9 张）输出尺寸严格为 1000×800 或 1000×1000。
- JPEG 可解码、方向正确且不携带 EXIF。
- 图片不得超出画布；间隔、字幕带和安全边距符合模板。
- 主、副文案在短名称和允许的最长名称下均不溢出。
- 动态中文与输入字段完全一致，不出现生成式伪文字。

### 12.3 API 与客户端测试

- 非法 variant 返回 400。
- 无效、过期、撤销或媒体摘要不匹配的 token 无法读取任何缓存封面。
- 好友 URL 返回 5:4，朋友圈 URL 返回 1:1。
- `onShareAppMessage` 与 `onShareTimeline` 使用不同 URL。
- 封面预取失败后使用正确比例兜底，不退回页面截图。
- 现有公开相册隐私、单媒体分享和视频播放测试继续通过。

### 12.4 微信验证

- 微信开发者工具分别验证好友/群聊卡片与朋友圈预览。
- 至少在一台 iOS 和一台 Android 真机验证：右侧照片不再被截出卡片、中文没有压线、长角色名可读。
- 手工覆盖 1、2、3、5、6、8、9 张强候选及“100 张相册只精选少量强图”的场景。

## 13. 发布策略

- 不需要数据库迁移。
- 缺省 `variant=friend` 保持旧封面 URL 兼容。
- 新旧小程序客户端可以同时访问部署后的服务端。
- 上线前先部署服务端和两张比例正确的兜底图，再发布使用双 URL 的小程序版本。

## 14. 完成标准

本设计完成的判断标准：

1. 任意 1～9 张最终精选照片均有明确布局。
2. 好友卡片严格 5:4，朋友圈严格 1:1。
3. 不会再出现整列照片超出微信分享卡片。
4. 大相册不会因为照片多而自动退化为九宫格；弱图和重复图会被舍弃。
5. 文案准确、有趣、可读，并且不泄露隐私。
6. 分享撤销、审核状态变化和隐私设置继续即时生效。
