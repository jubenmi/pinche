# 车局相册管理整改设计

日期：2026-06-29

## 结论

车局相册按“同车成员相册”理解。相册入口、照片列表、原图访问、隐私设置、上传和标注都只服务于发车后的同车成员；车头和管理员也不能绕过成员隐私查看不可见照片。

本次整改优先收紧隐私边界，再修正管理端按钮和小程序入口，最后用静态检查和 smoke 用例覆盖回归风险。

## 背景

当前相册已经具备发车后开放、成员上传、未标注照片只对上传者可见、被标注成员隐私保护等规则。但还有几处边界需要收口：

- 相册列表只要求登录和发车后开放，没有强制访问者是同车成员。
- 图片地址是短期签名 URL，但图片请求本身不需要登录凭证。
- 管理后台对所有可见照片都展示标注和删除按钮，和后端“仅上传者可操作”的规则不一致。
- 小程序非成员也能看到隐私设置入口，失败提示不够准确。

## 目标

1. 相册列表只允许发车后的同车成员访问。
2. 相册图片接口必须同时校验短期签名、登录用户和照片可见性。
3. 小程序和后台在展示图片时通过带登录态的受保护下载获取临时/对象 URL，不再依赖可被匿名打开的图片地址。
4. 后台只在当前用户有权限时展示标注和删除按钮。
5. 小程序只对同车成员展示隐私设置入口；403 提示明确说明成员和发车条件。
6. 检查脚本覆盖成员访问、图片授权和按钮权限。

## 非目标

- 不改变相册表结构。
- 不增加公开相册模式。
- 不允许系统管理员绕过成员隐私查看照片。
- 不做复杂审核流或相册运营后台。

## 核心规则

### 相册访问

`GET /api/sessions/:id/album` 必须满足：

- 用户已登录。
- 车局已到发车后开放窗口。
- 用户是车头、DM、NPC 或确认/锁定座位成员。

不满足成员条件时返回 403。

### 图片访问

`GET /api/session-album/photos/:id/image` 必须满足：

- 请求带有效登录 token。
- URL 带未过期的媒体签名。
- 登录用户是同车成员。
- 登录用户按上传者、被标注者、上传者隐私和被标注者隐私规则可见该照片。

图片响应使用私有缓存策略，避免把成员相册原图当作公开静态资源缓存。

### 前端展示

小程序通过带 Authorization 的受保护请求下载图片，再用本地临时文件展示和预览。

后台通过带 Authorization 的 `fetch` 获取图片 `Blob`，再用 `URL.createObjectURL` 展示和预览，并在刷新/卸载时释放对象 URL。

### 管理操作

标注和删除沿用“仅上传者可操作”的后端规则。前端必须按后端返回的 `can_tag` 和 `is_mine` 展示按钮；没有权限时不展示可点击操作。

## 验收

1. 非同车登录用户访问相册列表返回 403。
2. 未登录用户打开相册图片 URL 返回 401。
3. 同车但不可见该照片的用户打开图片 URL 返回 403。
4. 同车且可见该照片的用户可加载图片。
5. 后台别人上传的可见照片不展示标注/删除按钮。
6. 小程序非成员不会看到可用的隐私设置入口。
7. `scripts/d18-session-album-privacy-check.js` 通过。
8. `scripts/d18-session-album-privacy-smoke.js` 通过时覆盖成员相册和受保护图片。

## 自检

- 本设计没有扩大管理员查看能力。
- 本设计没有把相册照片恢复成公开静态文件。
- 图片 URL 即使被复制，也不能在没有登录态或没有可见权限的情况下直接访问。
- 前端展示方式和后端授权模型一致。
