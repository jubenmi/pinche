# 全项目北京时间统一设计

## 背景与根因

车局“野菩萨”的管理页显示 `2026-07-18 13:00`，日历列表却显示 `05:00`。API 返回的是表示同一时刻的 UTC ISO 字符串 `2026-07-18T05:00:00Z`。管理页通过 `Asia/Shanghai` 正确格式化为 `13:00`，但日历组件的解析器先用不含时区的正则匹配 ISO 字符串前缀，忽略末尾 `Z`，随后把 `05:00` 当作本地墙上时间构造，造成偏差 8 小时。管理后台的小程序预览还复制了同类解析逻辑。

## 目标

- 用户输入的业务日期时间一律按北京时间（`Asia/Shanghai`）理解。
- 小程序、管理后台、通知文案、日历分组和业务日期判断一律按北京时间显示或计算。
- 数据库存储和接口传输继续使用 UTC 标准时刻，避免服务器、容器或设备时区影响同一时刻的含义。
- 带 `Z` 或显式偏移量的时间必须作为绝对时刻解析，不能丢弃时区信息。
- 兼容历史无时区字符串：`YYYY-MM-DD HH:mm:ss` 按北京时间解释。

## 非目标

- 不迁移现有数据库时间列为北京时间墙上时间。
- 不修改只用于协议、安全签名、过期时间或遥测的 UTC 输出格式。
- 不进行与时间统一无关的页面重构。

## 方案

### 统一时间模块

在 `@pinche/shared` 增加无运行时环境依赖的时间工具，作为小程序、管理后台和 API 可复用的唯一规则来源。模块固定导出 `Asia/Shanghai` 业务时区，并提供以下能力：

- 解析绝对时间：接受 `Date`、ISO `Z`、ISO 显式偏移量。
- 解析历史墙上时间：无时区的数据库格式按北京时间解释。
- 格式化北京时间日期、时间和日期时间。
- 生成北京时间日期键，供日历分组使用。
- 将北京时间输入转换成 UTC ISO 字符串，供 API 传输。

解析规则必须先识别完整字符串是否带时区，再处理无时区格式；禁止用前缀正则吞掉 `Z` 或偏移量。

### 小程序

- `SessionCalendar.vue` 使用统一解析和北京时间格式化，不再调用设备本地的 `getHours()`、`getDate()` 生成业务展示和日期键。
- `sessionReschedule.js` 改为复用统一模块，保留现有改期校验和文案接口。
- 分享页、详情页和消息列表中直接截取或隐式解析时间的代码改用统一模块。
- 设备无论设置为哪个时区，同一个车局都显示同一北京时间。

### 管理后台

- 管理后台的小程序预览移除复制的 `parseMineStartAt` 逻辑，改用统一模块。
- 现有日期格式化函数统一显式指定 `Asia/Shanghai`，并逐步收敛到共享函数。
- 创建时间、更新时间、车局时间和相册时间均按北京时间展示。

### API 与数据库边界

- API 对外接受车局时间时要求 ISO 字符串带 `Z` 或显式时区偏移。
- API 内部比较使用时间戳或 `Date`，不使用格式化文本进行比较。
- MySQL 连接显式使用 UTC 会话时区，确保 `DATETIME` 的读写不受容器 `TZ` 影响。
- API 返回绝对时间时使用 UTC ISO；订阅消息和用户可见文案在输出前转换为北京时间。
- 数据库 `CURRENT_TIMESTAMP` 参与业务比较时必须与 UTC 存储约定一致。

## 数据流

1. 用户在北京时间选择 `2026-07-18 13:00`。
2. 客户端将其转换为 `2026-07-18T05:00:00.000Z` 后提交。
3. API 按绝对时刻校验并以 UTC 约定写入数据库。
4. API 返回带时区的 UTC ISO 时间。
5. 所有用户界面和通知通过统一模块显示 `2026-07-18 13:00`，日历日期键为 `2026-07-18`。

## 错误处理与兼容

- 无法解析的时间显示“时间待定”或现有页面占位文案，不猜测时区。
- 无时区历史值仅按北京时间兼容；新写入接口不得接受无时区值。
- API 遇到缺少时区的新请求继续返回明确的参数错误。
- 现有数据不批量改写，避免把正确的 UTC 时刻再次偏移。

## 测试与验收

- 单元测试覆盖 `2026-07-18T05:00:00Z`、`+08:00`、`-04:00` 和无时区历史值。
- 回归测试固定断言 `2026-07-18T05:00:00Z` 在日历、管理页、后台预览和通知中均显示 `2026-07-18 13:00`。
- 在非北京时间进程环境下运行时间工具测试，证明结果不依赖系统本地时区。
- 验证日期跨日，例如 `2026-07-17T16:30:00Z` 必须归入北京时间 `2026-07-18`。
- 运行小程序、管理后台和 API 相关测试及构建检查。
- 微信开发者工具中确认“野菩萨”的列表页和管理页同时显示 `13:00`。

## 发布与回滚

- 先发布共享模块及客户端修复，再发布 API 的显式 UTC 连接配置。
- 发布后检查生产 API 健康、数据库健康及车局列表/详情显示。
- 若出现兼容问题，客户端可回滚到旧构建；数据库无需回滚或恢复数据，因为方案不迁移现有时间值。
