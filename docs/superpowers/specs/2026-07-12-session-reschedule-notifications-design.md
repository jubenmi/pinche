# 车局改期与统一消息中心设计

日期：2026-07-12

## 目标

车头可以在车局开始前修改发车日期和时间。车上已有其他成员时，管理页必须二次确认。改期成功后，系统同时创建小程序内通知，并对所有已上车玩家和已绑定 NPC 工作人员尝试发送微信订阅消息。

本次同时把现有顶部“消息”面板升级为统一消息中心，使待审核申请、审核结果和车局改期共用同一个入口。

## 范围

### 包含

- 小程序车头管理页的改期入口、日期时间选择器和确认交互。
- 专用车局改期接口及服务端权限、时间、状态校验。
- 已上车成员和已绑定 NPC 工作人员的通知名单计算与去重。
- 车局改期微信订阅消息及订阅授权入口。
- 持久化小程序内通知、未读状态、通知列表和已读操作。
- 顶部消息面板统一展示待审核申请、审核结果和车局改期。
- 审核通过或拒绝后，同时创建小程序内审核结果通知。

### 不包含

- 要求成员同意后才能改期。
- 自动取消报名、释放座位或解除 NPC 绑定。
- 给待审核申请人发送改期通知。
- 发车开始后改回未来时间。
- 站内私信、聊天未读或通用营销通知。

## 核心业务规则

1. 只有车头或系统管理员可以改期。
2. 只有尚未开始的车局可以改期；服务端以当前时间和数据库中的 `start_at` 再次判断。
3. 新时间必须晚于服务端当前时间，且必须与原时间不同。
4. “其他已上车成员”包括：
   - 已确认或已锁定座位上的非车头用户；
   - 已绑定用户的 active NPC 工作人员角色。
5. 待审核申请不属于已上车成员，不触发二次确认，也不接收改期通知。
6. 没有其他已上车成员时可直接确认改期；存在其他已上车成员时，客户端必须展示原时间、新时间和成员数，并要求二次确认。
7. 专用接口也接收明确的成员确认标志。当服务器发现其他成员但请求未确认时，拒绝改期，避免绕过客户端确认。
8. 改期不改变座位、申请、角色、相册、聊天或车局状态。

## 管理页交互

车局总览中的“时间”使用本地化格式显示，不直接展示 ISO 字符串。发车前在时间行旁展示“改期”按钮；车局已开始后不展示入口。

点击“改期”后打开现有 TDesign 日期时间选择器：

- 初始值为当前车局时间；
- 最小可选时间为当前时间之后；
- 选择结果先停留在客户端，不立即更新车局；
- 新时间非法或没有变化时不给出有效提交。

提交时：

- 无其他成员：显示普通确认，确认后提交；
- 有其他成员：显示二次确认，文案包含成员数量、原时间和新时间，并提醒系统将通知车上成员；
- 用户取消：保持原时间，不调用接口；
- 提交期间禁用改期和其他管理动作，避免重复请求。

成功提示包含通知结果，例如“改期成功，已发送 3 条微信提醒；另有 1 人未订阅，可在小程序消息中查看”。失败时保留原时间和选择值，允许重试。

## 专用改期接口

新增专用路由：

`POST /api/sessions/:id/reschedule`

请求体：

```json
{
  "startAt": "2026-07-19T13:00:00+08:00",
  "membersConfirmed": true
}
```

服务端在事务中：

1. 锁定并读取车局，校验车头或管理员权限。
2. 校验原发车时间仍晚于当前时间。
3. 规范化并校验新时间晚于当前时间且不同于原时间。
4. 查询、去重通知接收者；车头本人排除在成员提醒之外。
5. 若存在其他成员且 `membersConfirmed` 不为 `true`，返回业务冲突，不更新。
6. 更新 `sessions.start_at`。
7. 为每位接收者写入一条持久化的 `session_rescheduled` 小程序内通知，包含车局、原时间和新时间。
8. 提交事务。

事务成功后逐一尝试发送微信订阅消息。微信发送属于尽力而为：未订阅、次数耗尽、模板未配置或单个发送失败都不回滚改期和站内通知。接口返回更新后的车局、接收人数，以及微信发送成功、跳过和失败数量。

## 微信订阅消息

新增“车局改期提醒”场景和模板配置。消息包含：

- 剧本或车局名称；
- 原发车时间；
- 新发车时间；
- 简短状态“车局已改期”；
- 跳转路径 `/pages/session/detail?id=<session_id>`。

成员必须在本人可触发订阅授权的用户动作后主动授权。直接上车的玩家可在上车成功动作后请求授权；经审核上车的玩家和由车头绑定的 NPC 工作人员，在本人进入车局详情后通过“改期提醒”操作请求授权。车头不能替其他用户授权。授权结果继续写入现有 `subscription_requests`，未授权不影响上车、绑定或改期。

## 统一小程序消息中心

新增持久化用户通知表。每条通知至少包含：

- 接收用户；
- 通知类型；
- 关联车局；
- 标题和摘要所需的结构化数据；
- 创建时间；
- 已读时间，可为空；
- 必填去重键，防止同一业务操作重复落消息。

第一期支持两类持久化通知：

- `signup_reviewed`：玩家的上车或 NPC 申请审核结果；
- `session_rescheduled`：已上车成员的车局改期。

现有车头“待审核申请”继续由车局实时状态计算，不复制成持久化通知。顶部消息徽标为“待处理申请数量 + 未读持久化通知数量”。

消息面板按时间和优先级统一展示：

- 待审核申请：展示待审数量，点击进入车头管理；
- 审核结果：展示通过或拒绝，点击进入车局详情；
- 车局改期：展示原时间到新时间，点击进入车局详情。

打开面板不批量标记已读。点击某条持久化通知时先标记已读，再跳转；若标记失败，仍允许跳转，之后可以重试同步。

## 数据一致性与失败处理

- 车局时间更新和小程序内改期通知在同一事务中完成，不能出现时间已改但站内通知缺失。
- 微信发送在事务外执行，避免外部网络调用长时间占用数据库事务。
- 接收者按用户 ID 去重；同一用户既占玩家座位又绑定 NPC 时只收到一条站内通知和一次微信发送尝试。
- 审核结果通知在审核事务中落站内消息，微信发送继续在事务提交后尽力而为。
- 重复提交同一次改期时，服务端通过“新时间与当前时间相同”拒绝，通知去重键作为额外保护。
- 客户端收到权限、已开车、时间非法或需要成员确认等错误时展示明确中文提示，不吞掉服务端原因。

## 接口边界

新增或调整以下接口：

- `POST /api/sessions/:id/reschedule`：专用改期。
- `GET /api/users/me/notifications`：分页读取当前用户持久化通知及未读数。
- `POST /api/users/me/notifications/:id/read`：将当前用户的一条通知标记已读。
- 现有“我的车局”读取继续提供待审核数量，供消息中心合并展示。
- 现有审核通过和拒绝服务增加站内审核结果通知，不改变原接口成功语义。

所有通知读取和已读接口只允许访问当前登录用户自己的记录。

## 测试策略

### 后端

- 非车头不能改期。
- 已开始车局不能改期。
- 新时间在过去或等于原时间时拒绝。
- 有其他成员但未确认时拒绝且无数据变化。
- 无其他成员时允许改期。
- 有玩家、NPC 或重复身份时正确计算并去重接收者。
- 改期和站内通知同事务成功；事务失败时二者都不落库。
- 微信通知失败不回滚改期，并正确汇总成功、跳过和失败数量。
- 审核通过和拒绝创建对应用户的站内通知。
- 通知列表和已读接口不能越权。

### 小程序

- 发车前显示改期入口，发车后隐藏。
- 日期时间选择和格式化正确。
- 有成员时必须二次确认，取消不发请求。
- 无成员时可直接确认提交。
- 成功后刷新页面时间并展示微信发送汇总。
- 顶部消息徽标正确合并待审核数和未读通知数。
- 三种消息类型展示正确并跳转到对应页面。
- 点击持久化通知后标记已读；打开面板本身不批量已读。
- 上车和 NPC 绑定成功后请求改期提醒订阅，不授权时原业务仍成功。

## 验收标准

1. 车头可在管理页把未开始车局改到一个未来时间。
2. 车上已有成员时，不经过二次确认无法改期。
3. 改期成功后，每位已上车玩家和已绑定 NPC 工作人员都有一条小程序内通知。
4. 系统对所有去重后的接收者尝试发送微信订阅消息，并向车头展示发送汇总。
5. 微信消息无法送达时，站内通知仍可在统一消息中心查看。
6. 审核结果和改期通知与现有待审核申请共用顶部“消息”入口。
7. 已开始车局无法通过页面或接口改期。
