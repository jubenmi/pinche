# 相册图片/视频单入口设计

## 状态

- 已于 2026-07-11 获得用户确认。
- 实施范围仅限微信小程序相册上传入口和选择结果分流。

## 背景

小程序车局相册当前有两个上传入口：主按钮通过 `uni.chooseImage` 选择图片，管理员专用次按钮通过 `wx.chooseMedia` 选择视频。图片和视频在选择之后分别进入已经稳定的压缩、校验、COS 上传与数据库登记链路。

本次把两个可见入口收敛为一个。管理员从同一个系统选择器中选择图片或视频，前端根据选择结果自动分流；普通成员仍只能选择图片。视频上传继续处于 `system_admin` 灰度阶段，后续放开权限时只调整可选媒体类型，不重写上传链路。

## 目标

1. 相册顶部只展示一个媒体上传入口，不再单独展示“短视频”按钮。
2. `system_admin` 通过同一入口选择图片或视频，并按选择类型复用现有上传链路。
3. 普通成员继续使用现有图片选择器，一次最多选择 9 张图片。
4. 保留现有图片压缩与大小限制，以及视频时长、压缩、大小限制和确认流程。
5. 保留前端管理员限制和服务端视频权限校验。
6. 不支持 `wx.chooseMedia` 时仍允许上传图片。

## 非目标

- 不向普通成员、车头、DM 或 NPC 开放视频上传。
- 不修改 API、COS intent、媒体登记接口或数据库结构。
- 不改变相册媒体查看、下载、标注、隐私和删除行为。
- 不增加多视频批量上传或图片与视频混合批量上传。
- 不修改管理后台 Web 的上传入口。

## 方案比较

### 方案 A：按权限混合兼容的单入口（采用）

- 管理员且当前微信支持 `wx.chooseMedia`：允许选择图片或视频，并根据返回类型分流。
- 普通成员：继续调用 `uni.chooseImage`。
- 管理员所在微信不支持 `wx.chooseMedia`：提示当前版本仅支持图片，然后回退到 `uni.chooseImage`。

该方案既满足管理员“选择后自动判断类型”，又不让普通成员和旧微信版本承担不必要的兼容风险。

### 方案 B：所有用户统一使用 `wx.chooseMedia`

根据角色动态传入 `mediaType`。代码表面更统一，但普通成员也会依赖较新的微信基础库，兼容收益不足，因此不采用。

### 方案 C：单按钮打开“图片/视频”操作菜单

可以原样复用两个选择器及各自的数量限制，但增加一次手动选择，不符合“选完文件后自动判断类型”的目标，因此不采用。

## 交互与布局

### 顶部操作区

- 主上传按钮保持现有角色/剧本标签、上传图标、主题色和禁用态。
- 主上传按钮和空相册上传按钮统一触发 `chooseAlbumMedia`。
- 删除管理员专用“短视频”按钮。
- 操作区恢复为“上传 + 隐私”两列布局，删除仅为三列视频布局服务的样式。

### 普通成员

点击上传入口后直接进入现有图片选择器，允许相册或相机来源，一次最多选择 9 张原图。选择成功后调用现有 `uploadChosenPhotos`。

### `system_admin`

支持 `wx.chooseMedia` 时调用统一选择器：

- `count: 9`；
- `mediaType: ["image", "video"]`；
- `sourceType: ["album", "camera"]`；
- 图片保持原图选择语义；
- 拍摄视频最长 60 秒，相册视频仍由上传前时长校验兜底。

选择成功后，根据每个临时文件的 `fileType` 判断类型；旧返回结构缺少 `fileType` 时使用结果级 `type` 作为兼容回退。

## 选择结果分流

新增一个无副作用的选择结果分类函数，使分流规则可以独立测试。输入为 `wx.chooseMedia` 的成功结果，输出仅允许以下三类：

1. `images`：全部文件均为图片，返回所有有效临时路径，调用 `uploadChosenPhotos`。
2. `video`：恰好一个文件且类型为视频，返回完整文件对象，调用 `uploadChosenVideo`。
3. `invalid`：空结果、未知类型、图片与视频混选或多个视频；不开始任何上传。

无效结果必须整体拒绝，不能先上传其中一部分。提示文案为：

- 混选或多个视频：`图片可多选，视频请一次只选 1 个`；
- 空结果或未知类型：`没有可上传的图片或视频`。

## 权限与兼容

- `canUploadVideo` 继续由 `!timelineMode && canUpload && isSystemAdmin` 决定。
- 普通成员的选择器参数不包含视频；前端不可选只是体验限制，服务端仍是最终权限边界。
- 视频上传 helper、视频创建接口及服务端 `system_admin` 校验不变。
- `wx.chooseMedia` 不可用时，管理员收到轻提示后回退到图片选择器，不阻断图片上传。
- `albumBusy` 检查保留在统一入口和现有上传方法中，防止重复选择和并发上传。

## 数据流

```text
统一上传按钮
  -> chooseAlbumMedia
      -> 普通成员 / chooseMedia 不可用
          -> choosePhotos
              -> uploadChosenPhotos
      -> system_admin + chooseMedia 可用
          -> classifyAlbumMediaSelection
              -> images -> uploadChosenPhotos
              -> video  -> uploadChosenVideo
              -> invalid -> toast，不上传
```

`uploadChosenPhotos` 与 `uploadChosenVideo` 之后的压缩、限制检查、COS 上传和媒体登记流程不变。

## 错误处理

- 用户取消系统选择器时沿用微信默认行为，不展示失败提示。
- 选择器返回无法识别的数据时展示轻提示，不修改上传状态。
- 图片或视频进入现有上传链路后，继续使用各自现有的状态文案和错误处理。
- 任何无效混选均不得产生 COS 对象或数据库记录。

## 测试与验收

### 自动测试

1. 先为选择结果分类函数编写失败测试，覆盖：
   - 多张图片；
   - 单个视频；
   - 结果级 `type` 兼容；
   - 图片/视频混选；
   - 多个视频；
   - 空结果和未知类型。
2. 更新小程序相册静态检查，锁定：
   - 两处上传 UI 都调用 `chooseAlbumMedia`；
   - 不再存在独立“短视频”按钮；
   - 管理员选择器同时声明图片和视频；
   - 图片和视频仍分别进入原上传方法。
3. 运行 D42 小程序视频回归、相册隐私/分享回归和小程序构建。

### 手工验收

1. 普通成员只看到一个上传入口，能一次选择并上传多张图片，不能选择视频。
2. `system_admin` 只看到一个上传入口；选择图片时走图片链路，选择单个视频时走视频压缩、确认与上传链路。
3. 管理员混选图片和视频或选择多个视频时收到明确提示，且没有任何文件开始上传。
4. 不支持 `wx.chooseMedia` 时仍可通过同一入口上传图片。
5. 上传期间按钮保持禁用，不能发起第二次选择。
