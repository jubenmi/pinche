# 微信小程序首页冷启动可见性修复设计

## 问题与证据

审核截图只显示微信导航栏，首页业务区域完全空白。使用与审核版本相同的生产构建做全新冷启动时，可稳定复现约 8–12 秒的纯白业务区；完成启动后，匿名公开车局接口正常返回并能渲染游客首页。

已经排除：

- 生产 `/health` 与匿名公开车局接口不可用；两者均直接返回 200。
- 微信 request 合法域名缺失；开启开发者工具域名校验后请求仍成功。
- 新 openid 无法创建用户；当前拒审已经不再出现 openid 提示，且游客首页在登录前就应可用。
- 单独升级基础库、关闭组件懒加载或把 COS 改成动态 import；这些实验都没有消除冷启动空白，动态 import 还会被构建器重新折叠进 vendor。

当前首页 WXML 的首批子节点全部是自定义组件：`AuthIdentityBar`、`TDesignFeedbackHost` 和 `SessionCalendar`。冷启动期间这些组件及公共 vendor 尚未就绪时，页面没有任何原生节点可显示，因此审核人员看到导航栏下方纯白。

## 已确认方案

首页启动态节点无条件存在，只通过动态内联 `display` 控制可见性；`isHomeReady` 只有在 `backendStatus.available !== null` 后才成立。在页面 JS 尚未注入任何数据、动态 `style` 仍为空时，基础样式会先显示由原生 `view` 构成的启动态。健康检查成功或失败后，内联 `display:none` 隐藏启动态，再挂载现有登录栏、反馈宿主、维护页或车局日历。

启动态不使用 TDesign、自定义组件、图片或新增网络请求，从而不依赖造成当前空窗期的组件链。现有登录、游客浏览、维护轮询、车局加载和分享逻辑保持不变。

## 边界

- 修改 `apps/miniprogram/src/pages/index/index.vue` 的首帧渲染分支与样式。
- 新增一个源码级冷启动回归测试，并接入现有检查命令。
- 不修改 API、openid、COS 上传、全局组件配置或业务路由。
- 不用“延长审核等待”或仅上传录屏掩盖白屏。

## 验收

1. 回归测试证明当前首页缺少原生启动态，并在修复后通过。
2. 生产构建成功；产物 WXML 的第一个节点必须是无 `wx:if`、仅带动态 `style` 的原生启动态，自定义首页组件位于后续 `wx:if` 分支。
3. 微信开发者工具清缓存、全新冷启动时，业务区不再出现纯白空窗，随后进入游客首页。
4. 验证登录入口与核心页面后，录制从冷启动到游客首页、登录及核心功能的完整审核视频。
