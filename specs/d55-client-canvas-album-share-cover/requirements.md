# D55 需求：客户端三图分享封面

## 1. 目标

公开相册分享封面改由微信小程序在本地 Canvas 生成。服务端不再读取封面候选原图、不进行清晰度或重复图分析、不渲染 JPEG，也不维护封面图片缓存或生成路由。

封面使用 1～3 张安全静态照片：有 1 张用单图、有 2 张用双图、有 3 张及以上只使用系统精选的 3 张。正文继续分页显示完整公开快照，封面数量不影响正文数量。

## 2. Requirement 1：安全三图配方

1. WHEN 创建新的公开相册分享 THEN 服务端 SHALL 按现有角色、审核、隐私、画面面积和时间规则从完整安全集合中确定最多 3 个封面媒体 ID。
2. WHEN 读取已有公开分享快照 THEN 服务端 SHALL 接受历史 `cover_media_ids` 中至多 30 个候选，但仅取其中排序靠前且仍安全的最多 3 项生成配方；不得因旧快照格式失效。
3. WHEN 返回公开相册 DTO 或创建分享 token DTO THEN 服务端 SHALL 返回相同结构的 `cover_recipe`，其中只包含最多 3 个静态照片的短期授权缩略图 URL、图片尺寸和可选焦点坐标；不得包含原图 URL、标签、上传者、隐私详情或未入选候选。
4. WHEN 配方中的媒体在分享创建后失效 THEN 服务端 SHALL 从配方中排除它；对应缩略图路由仍须继续做现有动态授权复核。
5. WHEN 没有可用的封面媒体 THEN 服务端 SHALL 返回空配方，客户端使用现有本地降级图。

## 3. Requirement 2：客户端 Canvas 封面

1. WHEN 小程序拿到有效 `cover_recipe` THEN SHALL 优先复用当前相册页面已持有的 recipe 媒体本地预览路径；仅在某张本地预览缺失时下载该 recipe 的 640px 缩略图兜底，并在本地 Canvas 生成好友 5:4 和朋友圈 1:1 两张分享图。
2. WHEN recipe 有 1、2 或 3 张图片 THEN SHALL 分别使用单图、双图和三图布局；最终画布不得使用第 4 张图片。
3. WHEN Canvas 生成成功 THEN `onShareAppMessage` 和 `onShareTimeline` SHALL 使用生成的本地临时文件路径作为 `imageUrl`。
4. WHEN 一个渠道的图片下载、Canvas 或临时文件导出失败 THEN SHALL 仅让该渠道回退至现有本地图，不阻塞另一渠道，也不得阻塞正文浏览。
5. WHEN 同一 share、配方或 token 已失效 THEN SHALL 丢弃旧的异步生成结果；页面刷新、token 变化、隐藏或卸载时 SHALL 清理本地封面状态。
6. WHEN 同一有效 share 在同一页面会话内重复准备封面 THEN SHALL 复用已生成的本地临时路径，不重复下载或绘制。

## 4. Requirement 3：删除服务器合成链路

1. 服务端 SHALL 删除公开封面图片路由、封面 capability token、实际图片分析、Sharp 拼图渲染、封面缓存、并发协调器和仅为该链路存在的依赖注入覆盖。
2. 服务端 SHALL 删除对应的单元、路由、渲染和静态门禁测试；保留用于公开相册图片缩略图、视频封面和正文分页的代码。
3. `cover_url` 与 `timeline_cover_url` SHALL 从公开 DTO 和分享 token DTO 中移除，并由 `cover_recipe` 取代。
4. 不新增数据库迁移；历史 `cover_media_ids` 和分享 token 语义保持可读。

## 5. 性能与验收

1. 服务端生成一次公开相册响应时不得读取任何封面图片对象、调用 Sharp 封面渲染或写入封面图片缓存。
2. 客户端每个渠道最多解析并绘制 3 张图片；优先使用本地预览，缓存未命中时最多下载 3 张 640px 缩略图，且不得请求封面原图。
3. API 测试覆盖新分享三图限制、旧 9/30 候选兼容、DTO 最小化和失效候选剔除；小程序测试覆盖 1/2/3 图布局、每渠道本地路径、缓存复用、过期结果关闭与单渠道降级。
4. 全量 `npm run check`、小程序构建和开发者工具刷新必须通过；`main`/`publish` 提升仅在更新后的 `develop` CI 成功后继续。
