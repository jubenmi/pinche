# D52 Requirements：历史车局时间纠错

更新日期：2026-07-22

版本：v1.0

状态：待产品复核

## 1. 目标与范围

D52 为已经到达开本时间的历史车局增加独立的时间纠错能力。当前车头可以把一个错误的历史时间修正为另一个已经过去的正确时间，用于修复早期缺陷或录入错误留下的历史数据。

历史纠错不是重新发车，也不是未来车局改期。现有 `POST /api/sessions/:id/reschedule` 继续只处理尚未开始的车局；D52 使用独立接口、独立校验和不可覆盖的审计记录。

## 2. 术语与业务边界

- **未来改期**：车局当前 `start_at` 晚于数据库当前时间，修改后时间也必须晚于当前时间。
- **历史纠错**：车局当前 `start_at` 已到达或早于数据库当前时间，修改后时间必须早于数据库当前时间。
- **当前车头**：`sessions.organizer_user_id` 对应的当前用户；车头转让后只有新车头可以纠错。
- **重新发车**：把历史车局时间改到未来。本期明确禁止。

如果数据库里的错误时间仍在未来，但车局实际上已经结束，车头端无法可靠证明历史事实，本期继续由管理员线下核验和修复。

## 3. 验收需求

### Requirement 1：历史车局提供独立纠错入口

**User Story：** 作为当前车头，我希望修正已经开过的车局时间，让历史详情、相册和游后感显示正确日期。

1. WHEN 车局 `start_at` 已到达或早于当前时间 THEN 车头管理页 SHALL 显示“纠正时间”。
2. WHEN 车局 `start_at` 晚于当前时间 THEN 管理页 SHALL 继续显示现有“改期”，SHALL NOT 显示“纠正时间”。
3. WHEN 用户选择历史纠错时间 THEN 选择器 SHALL 只允许选择已经过去的时间。
4. WHEN 用户提交前 THEN 页面 SHALL 二次展示原时间、新时间和“仅修正历史记录，不会重新发车”。
5. WHEN 纠错成功 THEN 页面 SHALL 刷新车局并展示最新时间。

### Requirement 2：只有当前车头可以执行

1. WHEN 未登录用户请求纠错 THEN API SHALL 返回 401。
2. WHEN 登录用户不是当前车头 THEN API SHALL 返回 403。
3. WHEN 车局不存在 THEN API SHALL 返回 404。
4. 系统管理员 SHALL NOT 通过车头自助接口绕过“当前车头”检查；异常数据仍走独立运维流程。
5. 权限判断 SHALL 在数据库事务内基于锁定后的最新车头执行，不能只相信客户端页面状态。

### Requirement 3：只允许过去时间修正为过去时间

1. WHEN 原 `start_at` 晚于数据库当前时间 THEN 历史纠错 API SHALL 返回 409，提示使用未来改期流程。
2. WHEN 新 `startAt` 等于或晚于数据库当前时间 THEN API SHALL 返回 400，不能把历史车局重新变成未来车局。
3. WHEN 新时间与原时间在数据库秒级精度下相同 THEN API SHALL 返回 400。
4. WHEN `startAt` 缺少明确时区、日期无效或时区偏移非法 THEN API SHALL 返回 400。
5. 时间解析 SHALL 接受合法 ISO-8601 `Z` 或显式偏移，并统一为数据库秒级精度。
6. 服务端 SHALL 使用数据库当前时间作为车局是否已经开始的最终依据。

### Requirement 4：原子更新与不可覆盖审计

1. WHEN 纠错成功 THEN `sessions.start_at` 更新和审计记录插入 SHALL 在同一事务内完成。
2. 每条审计 SHALL 保存车局 ID、操作车头 ID、原时间、修正后时间和操作时间。
3. WHEN 任一步骤失败 THEN 时间和审计 SHALL 一起回滚。
4. WHEN 连续纠错多次 THEN 系统 SHALL 逐次追加审计，SHALL NOT 覆盖已有记录。
5. WHEN 并发请求修改同一车局 THEN 行锁 SHALL 保证后一个请求基于前一个请求提交后的最新时间校验和记录。

### Requirement 5：纠错不产生其他业务副作用

1. WHEN 历史时间纠错成功 THEN 车局状态、可见性、座位、报名、成员、相册、游后感和聊天数据 SHALL 保持不变。
2. 历史纠错 SHALL NOT 调用普通改期通知、微信订阅消息或站内改期消息。
3. 历史纠错 SHALL NOT 重新开放报名、改变审核规则或生成新车局。
4. 依赖 `sessions.start_at` 展示历史日期的现有页面 SHALL 在重新读取后自然显示修正值。

### Requirement 6：稳定的错误反馈

1. 原时间仍在未来时，客户端 SHALL 提示“这辆车还没有开始，请使用改期”。
2. 新时间不是过去时间时，客户端 SHALL 提示“历史纠错只能选择已经过去的时间”。
3. 新旧时间相同时，客户端 SHALL 提示“新时间与原时间相同，请重新选择”。
4. 权限变化、车局不存在或服务异常时，客户端 SHALL 保留当前页面并展示可理解提示。
5. WHEN 服务端判定生命周期或权限已变化 THEN 客户端 SHALL 重新加载车局，避免继续展示过期操作入口。

### Requirement 7：验证与回归

1. D52 SHALL 先建立失败测试或失败契约，再写最小实现。
2. 服务端单元测试 SHALL 覆盖合法时区归一化、非法时间、未来原时间、未来目标时间、相同时间、非车头和成功事务。
3. 成功事务测试 SHALL 验证只更新 `sessions.start_at`、追加一条审计且不调用任何通知能力。
4. 小程序单元测试 SHALL 覆盖入口互斥、历史时间校验、确认文案和错误提示。
5. 静态契约 SHALL 锁定迁移、独立路由、管理页入口和现有改期接口未被放宽。
6. 最终实现 SHALL 通过 D52 定向测试、现有 session reschedule 回归、`npm run check` 和 `npm run build:mp-weixin`。

## 4. 非目标

- 不允许把历史车局改到未来。
- 不处理“数据库时间仍在未来、实际已经开本”的自助纠错。
- 不修改剧本、店家、DM/NPC、押金、备注、座位或价格。
- 不增加审计历史列表、撤销按钮或管理员后台。
- 不发送微信订阅消息或站内通知。
- 不批量迁移历史错误数据，也不自动推断正确时间。
